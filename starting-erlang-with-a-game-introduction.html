Erlang, Functional Programming and the Actor Model.

Ok... so now, why then, Functional Programming?

Let me go on a tangent here. Having worked on some project that got complicated fast and just exploded with more and more code paths as the rules and 'special rules' for the business logic kept expanding without bound (the world just wants to add things in, never taking things out) I began to wonder to myself, is there any better way for me to structure and organize my code, with these desired characteristics -

  • Reading: That code would be so easy to read, that at a glance, you would know which part does what and where to change, as easy as reading HTML or even plaintext.
  • Writing: As easy as writing a blog. As easy as writing HTML or even plaintext,
  • Maintenance/updating: Where you would just know, at a glance, that changing a certain part of code that you see in front of you will not cause some propagated unknown change in some distant part of the code... And you didn't need to delve in and trace, all possible code paths deep into the code...
  • That you could be swap in/out entire blocks of code with well-defined purpose when you want to change rules/its behaviour.

For these goals, it would be good if you could write your code in a declarative manner, much like HTML, rather than an imperative manner, where you needed to trace line by line your code. For declarative, you should be able to at a glance, know 'what just is'.

It always inspired me that some people can do a one-man show, write code, and build an entire company out of it. It's probably still possible today (and people are already/still doing it) but until I would like to reach that level myself one day soon.1

Ok, given that the above desired characteristics are probably more of code organization than inherent any programming language itself, still there are perhaps some languages which happen to encourage/enforce writing in such a manner more than others.

Back to Functional Programming. The buzz about it today is that programs structured thus are more scalable, maintainable, and a lot of buzzwords like 'no side effects', 'no mutable state', 'lambda calculus', etc. etc. which we probably don't really need to spew off to be able to begin writing a program.

The thing we need to know is a 'basic unit' of such programs is a function which takes in certain inputs and returns an output, and in the body of the function, it doesn't depend on any other state, solely on the inputs.

  • That makes splitting such processing over any number of processing units/cores feasible,
  • The output is always the same for a given set of inputs no matter the state of anything else, that makes things more easily repeatable and debuggable

Anyway, reading the first paragraphs, it is exactly the kind of situation that makes me search for better ways to do things. I paste it here for my own reference and inspiration just in case that page gets taken off unexpectedly one day.

As programs get bigger, they also become more complex and harder to understand. We all think ourselves pretty clever, of course, but we are mere human beings, and even a moderate amount of chaos tends to baffle us. And then it all goes downhill. Working on something you do not really understand is a bit like cutting random wires on those time-activated bombs they always have in movies. If you are lucky, you might get the right one ― especially if you are the hero of the movie and strike a suitably dramatic pose ― but there is always the possibility of blowing everything up.

Admittedly, in most cases, breaking a program does not cause any large explosions. But when a program, by someone's ignorant tinkering, has degenerated into a ramshackle mass of errors, reshaping it into something sensible is a terrible labour ― sometimes you might just as well start over.

Thus, the programmer is always looking for ways to keep the complexity of his programs as low as possible. An important way to do this is to try and make code more abstract. When writing a program, it is easy to get sidetracked into small details at every point. You come across some little issue, and you deal with it, and then proceed to the next little problem, and so on. This makes the code read like a grandmother's tale.

What we're talking about there is we want to make our code declarative - to me, something like HTML is the best, as we know what it is at a glance - no need to trace.
